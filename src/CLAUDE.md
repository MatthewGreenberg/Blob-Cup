# src

- **Tournament / menu** — `App.jsx` holds `screen` (`menu|about|bracket|match|result|champion`), `mode` (`practice|tournament`) and `round` (0–2); `<Ui />` (DOM overlays, `App.css` `.ui-*`/`.bracket-*`) renders the home menu (Tournament / Practice / About), an animated 8-team scripted bracket (Player 1's road: Red Blobs → Green Blobs → Bears; other matchups are flavor with scripted winners revealed round by round — a stage's lineup shows `???` until you reach it and losers are only struck through once you're past it, no spoilers; stage labels sit in an aligned header row above a `.bracket-matches` space-around flex; columns stagger in via `--d` animation-delay, the current matchup pulses), the in-match HUD (broadcast-style, `.ui-hud-*` in `App.css`): a single cohesive top-left glass panel (`.ui-hud-panel`, `overflow:hidden` so the colored score row is full-bleed) stacking three bands — a header row (`.ui-hud-panel-head`: stage label left, `X leads`/`All square` status right), a split scoreboard row (blue Player 1 goals / round-tinted opponent saves — `OPP_THEME[round]` classes: Red default, `is-green`, `is-yellow` for Bears — `VS` divider, circular team-icon slots on the outer ends, currently blank placeholders awaiting real team icons), and a `Shot` tracker row (5 pills, each = numbered circle + a mark badge below; current shot's circle is blue-filled, taken shots show a green ✓/red ✕ from `match.log` — a per-shot goal/save history tracked alongside goals/saves) — plus a top-right glass `Menu` button (hamburger glyph) that opens a `paused` overlay — `Back to Main Menu` + `Resume` always, plus `Restart Tournament` (round 0 → bracket) in tournament mode. The panel lives top-left (not bottom-center) so it clears the bottom-center power bar. In tournament mode `<Hud />` hides its own `.game-score` pill and `.game-hint` tutorial line (`mode` prop) — the shot tracker replaces the pill and the tutorial is redundant; the power bar still shows. A FIFA-style round-intro stripe (`.ui-roundintro`, mounted for 2.6s by `startMatch(nextMode)` — mode passed explicitly since the prop is stale in the same click), and result (big `.ui-scoreline` digits)/champion screens. Overlay titles use `ui-pop`, NOT the HUD's `banner-pop` (those keyframes bake in the banner's `translate(-50%,-50%)` and would shift centered-flow titles half their size left/up). Full-screen overlays have `pointer-events: auto`, which is also what blocks Game's canvas input between matches — Game itself was not taught about screens. Match rule: first to `MATCH_GOALS_TO_WIN=3` goals out of `MATCH_SHOTS=5` (loss at 3 saves); Ui counts `stadium:goal/save` in a ref and flips to the result screen on the following `stadium:reset` so celebrations play out. Win → next round or champion; loss → retry same round. The active round config `cfg` (from `ROUNDS`/`PRACTICE_CFG` in `game/constants.js`) prop-drills App → Scene → Stadium → Game: `keeper.{readChance,reachX,reachY,react,scale}` override the KEEPER_* baselines in Game's `onUp` (effect re-binds on `cfg`), so the keeper gets harder each round; `cfg.{charge,perfectMin,perfectMax}` (default to `CHARGE_TIME`/`PERFECT_*` via `??`) tune the power bar per round — later rounds ping-pong faster and shrink the gold zone (0.9s/0.09-wide → 0.8s/0.075 → 0.72s/0.075; the Bears are the final boss — near-perfect read (0.95), fastest react, `safePower: 0.62` (anything short of a strong kick is auto-caught) and `bendGood: 2.1` (only a near-max bend fools him), with `reachX: 1.35` deliberately under the corner gap (aim ±6.1 − dive clamp ±4.6 = 1.5) so a full-power corner ball can still score — his only weaknesses are the perfect band, real pace to a corner, or big bend + pace; plus the fastest ping-pong and tightest zone); Game reads them in `useFrame`/`onUp` and `<Hud cfg>` positions the gold band from them; Practice = the easiest keeper, the red blob (`PRACTICE_CFG` mirrors `ROUNDS[0]`, tint red / `goalie: 'blob'`), so newcomers can score. `cfg.goalie`: rounds 1–2 use a **blob goalie** — one giant `fan_blob.glb` mesh (vertexColors + glossy swap) tinted `cfg.tint` red/green via `material.color`; it has no rig, so dives are procedural in Game's `useFrame` (idle bounce, topple-lean `rotation.z` toward `sign(diveX)`, squash on central blocks — the keeper group's existing `position.x` slide carries it). The bear stays mounted but `visible=false` during blob rounds so its animation bindings/mixer keep working untouched (clips play on the hidden bear, harmlessly). `<Hud />` only mounts while `screen === 'match'`, so per-match score reset is just remounting.

- **Loading / entrance** — `Loading.jsx` (DOM overlay in App.jsx, `.loader-*` in `App.css`) is pure DOM/CSS — no second WebGL canvas: a CSS-keyframed bouncing soccer ball with squash (pattern = inline SVG data URI in `.loader-ball`, cream circle + clipped pentagon patches, highlight gradient layered on top; + synced shadow ellipse), a candy-pink progress bar, and a "loading N%" label. Progress = drei `useProgress` (the zustand hook works outside a Canvas); the displayed pct is a **running max** (`if (pct > shown) setShown(pct)` render-adjust — raw progress dips when the env map joins the load queue mid-boot). It exists because the main R3F canvas doesn't paint a single frame until its Suspense tree resolves; CSS animation keeps the bounce smooth while the main thread parses GLBs. Sequence: `Stadium` mounts (Suspense resolved) → emits `stadium:loaded` → Loading fades 750ms (matching `.loader` CSS transition) and unmounts; CameraRig's ready latch flips on the same event and the normal damping flies the intro aerial down into the menu shot; `Ui.jsx` reveals the menu 750ms after `stadium:loaded` (`entered` state — the delay equals the loader fade, so the menu stagger starts the instant the overlay is gone, over the camera fly-in). Gotcha fixed 2026-07: a stray duplicate `.loader` glass-card rule later in `App.css` used to clobber the overlay's background — don't reintroduce the class name for other UI.

- **Mobile / touch (2026-07)** — the game is fully touch-playable. Input: touch has no hover, so Game's `onDown` raycasts the actual press point (module-scope `POINTER_NDC` scratch, same plane/clamp math as the useFrame reticle follow) and snaps the reticle there before locking `shot.aim` — tap where you want to aim, hold to charge, drag to curl, release; the curl drag rides R3F's `state.pointer`, which keeps updating because the canvas has `touch-action: none` (`.stadium-canvas canvas` in App.css — without it the browser steals the drag as a scroll and fires pointercancel = instant weak shot). `index.html` viewport adds `maximum-scale=1, user-scalable=no, viewport-fit=cover` (double-tap zoom would fight the charge gesture); `index.css` body gets `overscroll-behavior: none` + `user-select: none` + `-webkit-touch-callout: none` (long-press was popping selection/callouts mid-charge). Notch/home-indicator safety: every corner-pinned widget (`.ui-hud-panel`, `.ui-hud-menu`, `.sound-btn`, `.game-score`, `.game-hint`, `.game-power` bottom) wraps its offset in `max(clamp(...), env(safe-area-inset-*))`, and `.ui-overlay` pads with safe-area insets + gets `overflow-y: auto` and `justify-content: safe center` (fallback line after plain `center`) so short/landscape phones scroll instead of clipping overlay tops. Camera: the `SHOTS` are framed for landscape and three.js fov is vertical, so portrait cropped the sides to a tunnel — `aspectFov(fov, aspect)` in Scene.jsx (applied at CameraRig's fov damp, aspect from `state.size`) preserves the horizontal frustum below `REF_ASPECT=1.6` (widening/zooming out every shot on phones), capped at `FOV_CAP=68` before wide-angle distortion. The `sway===0` play-shot pointer mouse-look is disabled on coarse pointers (`COARSE_POINTER = matchMedia('(pointer: coarse)')` in Scene.jsx) so touching to aim/charge doesn't swing the camera. The power ping-pong runs 1.4× slower on coarse pointers (`CHARGE_MULT` in Game.jsx, folded into the charge divisor) so the gold zone is catchable with a touch. Readability: `.ui-hud-lead` bumped to 11px, stage label + team names ellipsize (`min-width: 0` on `.ui-hud-panel-head`/`.ui-hud-team`), and a widened 680px media query shrinks the in-match HUD (panel `calc(100vw - 110px)` so it clears the Menu button, 22px icons, 26px pills), bracket cards (128px), buttons and about text for phones.

- **First-visit tutorial (2026-07)** — `<Tutorial />` (`components/Tutorial.jsx`, DOM overlay mounted next to `<Hud />` while `screen === 'match'`, `.tutorial-*` in `App.css`): a 3-step coach card teaching the bend mechanic, gated by `localStorage['stadium:tutorial'] = 'done'` (`game/tutorial.js`: `tutorialDone()`/`markTutorialDone()` + the module-scope `TUTORIAL.freeze` flag the Canvas components read in useFrame — same no-prop-drilling pattern as sound.js's `muted`). Steps wait on real game events, and the game literally stops at each beat — `TUTORIAL.freeze` halts **gameplay time**: the power ping-pong, the hold clock (`shot.t`, so the release windup math stays consistent), the player's run-up (walk translation + mixer `timeScale=0` — statue mid-stride; mixers are mutated via refs seeded in `useLayoutEffect`, `mixerRef`/`keeperMixerRef`, because writing a hook return in useFrame trips react-hooks/immutability), the keeper's mixer and the blob keeper's idle bounce, while ambient layers (crowd, flags, jumbotron, atmosphere) keep living; a `.tutorial-dim` radial vignette overlays the stopped beats (aim + bend steps) to spotlight the card: (1) *aim* — game idles in `aim` phase until the press (`stadium:windup` advances, freeze on); (2) *bend* — frozen mid-charge until the drag has crossed `KEEPER_BEND_GOOD` in **both directions** — Game emits `stadium:bend` with detail ±1 once per charge per side (`shot.bendPingedL/R`, reset on pointerdown); the card tracks `bent {left,right}` state (◦/✓ progress in the sub line + "now bend it the other way" copy, plus the "bending makes it much harder for the goalie" tip), then lingers `BEND_DWELL=1.6s` with the meter still frozen before advancing; while frozen, Game's `onUp` **cancels instead of kicking** (back to `aim` + `stadium:reset` — reversing the drag makes accidental lifts easy, and a release would fire a weak kick past the lesson; Hud also drops the power bar on reset since no kick ever fires); (3) *release* — meter resumes at **half speed** (`TUTORIAL.slow`, set when the bend dwell ends, cleared on kick/skip/unmount; Game doubles the effective charge time in its ping-pong) so first-timers can read the gold zone, `stadium:kick` swaps in a ✓ *done* card that hangs `DONE_LINGER=2.4s` and persists. Kicking without completing both bends restarts at step 1 on `stadium:reset` (done card is exempt); a `skip` button (stopPropagation like SoundToggle) and a component-unmount cleanup both clear the freeze + timer so the meter can never stay stuck. Hud's practice `.game-hint` line only renders once `tutorialDone()` so the two bottom-center messages don't stack. On phones (≤680px) the `.tutorial-card` becomes a full-width bar pinned to the bottom edge (rounded top corners, top border only, safe-area bottom padding, slides up via a media-query-local `tutorial-pop` keyframe override) instead of the floating desktop card.

- **Sound** — `game/sound.js`: two looped beds stay HTMLAudio at module scope — `bg-loop.mp3` (30% vol) and `crowd-noise.mp3` (20% vol) — but one-shots `kick.mp3`/`cheer.mp3` play through **Web Audio** (module-scope `AudioContext`, buffers fetched+decoded at load): Safari's `HTMLAudio.play()` has 100–300ms latency and `currentTime=0` stalls, buffer sources fire sample-accurately so the kick lands on foot-contact. The cheer's tail fade-out is a `linearRampToValueAtTime` gain ramp (no setInterval). Browsers need a user gesture to unlock audio, so App's effect calls `unlockSound()` on the first window `pointerdown` (starts the loops + `ctx.resume()` — the context is born suspended) and plays `playKick()` on `stadium:launch` (foot-on-ball) and `playCheer()` on `stadium:goal`. A shared module `muted` flag; `toggleSound()` flips it (pausing/resuming the beds) and returns the new "on" state. `<SoundToggle />` (bottom-left glass button, `.sound-btn` in `App.css`) drives it — animated SVG soundwave when on, flat line when muted; it `stopPropagation`s all pointer events so clicks don't reach Game's window pointerup/down handlers. On phones (≤680px) `.sound-btn` is hidden (it collided with the bottom tutorial bar) and the mute control moves into Ui's pause menu as a `.ui-btn--mobile` **Mute Audio / Unmute Audio** button (hidden on desktop, shown only in the 680px query; Ui mirrors the on/off state via `toggleSound()`'s return).

Re-run after a Blender re-export: `npx @gltf-transform/cli optimize public/<name>.glb public/<name>.glb --texture-compress webp --texture-size 2048 --compress draco --instance false --simplify false`.
